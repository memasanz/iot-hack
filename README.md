# IoT Telemetry Simulator

A containerized Python application that generates synthetic telemetry data from simulated industrial field devices and streams it to **Azure IoT Hub** using the IoT device SDK. The Container App authenticates via **managed identity** — no connection strings needed. Includes Bicep templates and deployment scripts for a one-command deploy to Azure.

## API Endpoints

The simulator exposes a FastAPI server on port **8000** for controlling the simulation at runtime. The simulation auto-starts on container startup.

| Method | Endpoint | Description |
|---|---|---|
| `GET` | `/health` | Health check — returns `healthy` and whether simulation is running |
| `GET` | `/status` | Detailed status: device counts, cycle count, events sent, anomalies |
| `POST` | `/start` | Start the simulation (409 if already running) |
| `POST` | `/stop` | Stop the simulation (409 if already stopped) |
| `GET` | `/docs` | Interactive Swagger UI (auto-generated by FastAPI) |

### Example usage

```bash
# Check status
curl https://<your-app>.azurecontainerapps.io/status

# Stop simulation
curl -X POST https://<your-app>.azurecontainerapps.io/stop

# Start simulation
curl -X POST https://<your-app>.azurecontainerapps.io/start
```

## Device Types

| Sensor Type | Unit | Normal Range | Example Locations |
|---|---|---|---|
| Temperature | °C | 18–85 | Furnace A, Cooling Tower, Server Room |
| Pressure | PSI | 30–120 | Hydraulic Press, Boiler Room, Steam Line |
| Vibration | mm/s | 0.5–10 | Motor A, Pump Station, Conveyor Belt |
| Humidity | %RH | 30–70 | Clean Room, Storage Vault, Electronics Lab |

## Anomaly Patterns

The simulator injects realistic anomalies at a configurable probability:

- **Spike** — sudden jump to extreme values (e.g., temperature shoots to 130°C)
- **Drift** — gradual sensor offset that accumulates over time
- **Sensor failure** — device goes offline for 2–6 cycles, reporting `null` values

## Telemetry Schema

Each event is a JSON object sent as a device-to-cloud (D2C) message to IoT Hub:

```json

  "firmware_version": "2.3.15",

```

## Deploy to Azure

### Prerequisites

- [Azure CLI](https://learn.microsoft.com/cli/azure/install-azure-cli) installed and logged in (`az login`)
- An Azure subscription

### One-command deploy

Deployment is split into two steps to support **separation of duties**. In many organizations, the team that provisions infrastructure (Contributor) is not the same team that manages access control (Owner / User Access Administrator). By separating resource creation from role assignment, a developer can deploy the application while an admin reviews and approves the privileged role assignment independently.

#### Step 1 — Deploy resources (requires **Contributor**)

Creates all Azure resources (ACR, IoT Hub, Container App) and builds the container image:

**PowerShell:**
```powershell
.\infra\deploy.ps1 -ResourceGroup "mbi-iot-rg" -Location "eastus" -Prefix "mmxiot" -DeviceCount 20 -ConsumerGroupCount 3
```

**Bash:**
```bash
RESOURCE_GROUP=mbi-iot-rg LOCATION=eastus PREFIX=mmxiot DEVICE_COUNT=20 CONSUMER_GROUP_COUNT=3 bash infra/deploy.sh
```

#### Step 2 — Assign roles (requires **Owner** or **User Access Administrator**)

Grants the Container App's managed identity the IoT Hub Data Contributor role:

**PowerShell:**
```powershell
.\infra\deploy-roles.ps1 -ResourceGroup "mbi-iot-rg" -Prefix "mmxiot"
```

**Bash:**
```bash
RESOURCE_GROUP=mbi-iot-rg PREFIX=mmxiot bash infra/deploy-roles.sh
```

> **Note:** If the deploying user has Owner access, both steps can be run back-to-back. The role assignment script auto-discovers the IoT Hub name and Container App principal ID from the prefix.

### What gets deployed

| Resource | Description |
|---|---|
| **Azure Container Registry** | Hosts the simulator Docker image |
| **Azure IoT Hub (S1)** | Receives D2C telemetry; devices auto-registered via managed identity |
| **Consumer Groups** | Configurable number (default 2), named `team1`, `team2`, etc. |
| **Container App Environment** | Managed environment with Log Analytics for monitoring |
| **Container App** | Runs the simulator with system-assigned managed identity |
| **Role Assignment** | IoT Hub Data Contributor — allows the Container App to register and manage devices |

### Deploy parameters

| Parameter (PS / Bash) | Default | Description |
|---|---|---|
| `Prefix` / `PREFIX` | `mbiiot` | Prefix for all resource names (must be globally unique for ACR/IoT Hub) |
| `Location` / `LOCATION` | `eastus` | Azure region |
| `DeviceCount` / `DEVICE_COUNT` | `10` | Number of simulated devices |
| `SendInterval` / `SEND_INTERVAL` | `5` | Seconds between telemetry cycles |
| `AnomalyProb` / `ANOMALY_PROB` | `0.05` | Anomaly probability (0–1) |
| `ConsumerGroupCount` / `CONSUMER_GROUP_COUNT` | `2` | Number of consumer groups to create |

### Authentication

The simulator uses **managed identity** throughout — no secrets or connection strings are stored:

- The Container App's **system-assigned managed identity** is granted **IoT Hub Data Contributor** on the IoT Hub
- On startup, the simulator uses the service SDK to register each simulated device in IoT Hub
- Each device then connects using its own SAS key (auto-generated by IoT Hub)

### Connect downstream consumers

The IoT Hub built-in Event Hub-compatible endpoint can be consumed by multiple teams in parallel using separate consumer groups:

1. **Fabric Real-Time Hub** — Add Azure IoT Hub as a source in Fabric. See the step-by-step guide: [Add Azure IoT Hub as source in Real-Time Hub](https://learn.microsoft.com/en-us/fabric/real-time-hub/add-source-azure-iot-hub)
2. **Azure Data Explorer** — Create a data connection with its own consumer group

Each consumer group reads independently — one team's consumption does not affect another.

### Monitor the simulator

```bash
# Stream live logs
az containerapp logs show -n <prefix>-simulator -g <resource-group> --follow

# Check running status
az containerapp show -n <prefix>-simulator -g <resource-group> --query "properties.runningStatus"
```

## Local Development

### Run locally (dry-run mode)

Without an `IOT_HUB_HOSTNAME`, the simulator prints telemetry to the console:

```bash
pip install -r requirements.txt
python -m app.main
```

### Run with Docker Compose

```bash
cp .env.example .env
# Edit .env with your IoT Hub hostname (or leave empty for dry-run)
docker compose up --build
```

## Environment Variables

| Variable | Default | Description |
|---|---|---|
| `IOT_HUB_HOSTNAME` | *(empty = dry-run)* | IoT Hub hostname (e.g., `myhub.azure-devices.net`) |
| `DEVICE_COUNT` | `10` | Number of simulated devices |
| `SEND_INTERVAL_SECONDS` | `5` | Seconds between telemetry cycles |
| `ANOMALY_PROBABILITY` | `0.05` | Probability (0–1) of anomaly per reading |
| `LOG_LEVEL` | `INFO` | Logging level (DEBUG, INFO, WARNING, ERROR) |
